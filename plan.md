# План: Интеграция Hermes с Telegram (MTProto) и Google Calendar (MCP)

## Цель
Реализовать интеграцию Hermes с Telegram (через MTProto user session для полного доступа) и Google Calendar через MCP серверы. Настроить автоматический парсинг событий из ВСЕХ чатов (со списком исключений) в календарь. Парсинг должен извлекать даты и время не только из чисел, но и из контекстных слов и относительных дат ("завтра", "в три"). Модерация спама временно отложена.

## Текущий контекст и выбранные компоненты
Поиск по GitHub выявил следующие популярные MCP-серверы:
- **Telegram (MTProto/Bot):**
  - `chigwell/telegram-mcp` (1086 stars) - кандидат для интеграции с Telegram. *Примечание: проверить поддержку MTProto user sessions (telethon/pyrogram).*
  - `moltis-org/moltis` (2675 stars) - шире, чем просто Telegram.
- **Google Calendar:**
  - `taylorwilsdon/google_workspace_mcp` (2389 stars) - кандидат для работы с Google Workspace.

## Ограничения
- **Telegram:** Только MTProto (без Browser/Bot API/Desktop Export).
- **Парсинг чатов:** Анализировать ВСЕ чаты, кроме явно указанных в списке исключений.
- **Поиск событий:** Способность парсить относительные даты (например, "завтра позвонит коллега в три" -> 15:00 следующего дня).
- **Крон и модели:**
  - Парсинг по расписанию (cron) новых сообщений.
  - Использовать бесплатные модели для парсинга (для снижения затрат на LLM-анализ всех сообщений).
  - Ротация бесплатных моделей для обхода rate limits и fallback на другие бесплатные модели.
- **События:** Не создавать дубликаты.
- **Секреты:** Не хранить в коде/git.
- **Действия:** Hermes не отправляет сообщения от имени пользователя (только системные подтверждения, если настроено).
- **Модерация:** Функционал поиска спама пока исключен.

## Предлагаемый подход и пошаговый план

### Этап 1: Аудит и подготовка окружения
1. Проверить установленные MCP серверы в Hermes (`hermes mcp list` / `hermes mcp config`).
2. Создать директорию для конфигураций (например, `~/.hermes/configs/tg_cal_integration`).

### Этап 2: Установка и настройка MCP-серверов
1. **Google Calendar MCP:**
   - Клонировать/установить `taylorwilsdon/google_workspace_mcp`.
   - Настроить OAuth credentials.
   - Интегрировать сервер в конфигурацию Hermes.
   - Настроить allowlist для tools.
2. **Telegram MTProto MCP:**
   - Изучить `chigwell/telegram-mcp` или написать кастомный сервер на Python (Pyrogram/Telethon).
   - Настроить авторизацию (api_id, api_hash, session string).
   - Настроить чтение сообщений из всех чатов (и получение списка чатов для исключений).

### Этап 3: Создание конфигурации и состояния
1. Создать файл `tg_cal_config.yaml`:
   - `excluded_chats`: [...] # список чатов-исключений
   - `llm_rotation_pool`: [...] # список бесплатных моделей (напр., gemini-1.5-flash, llama-3, qwen) для обхода rate limits
   - `parsing_prompt`: "..." # системный промпт с правилами парсинга относительных дат (завтра, через час, в понедельник)
2. Инициализировать `state.db` для:
   - `processed_message_ids_per_chat` (отметка "прочитано до")
   - `calendar_event_links`
   - `errors`

### Этап 4: Разработка логики процессора (Hermes Native Cronjob)
1. **Health/Status команда:** Скрипт `check_health.py`.
2. **Процессинг сообщений (встроенный Hermes `cronjob`):**
   - Создать периодическую задачу через `hermes_tools` (или команду `cronjob action='create'`).
   - Получить список всех чатов, отфильтровать по `excluded_chats`.
   - Для каждого чата получить новые сообщения (с момента последнего `processed_message_id`).
   - **Анализ текста:**
     - Выбрать бесплатную модель из пула ротации (в Hermes можно переопределять `provider`/`model` в параметрах, либо реализовать логику ротации внутри исполняемого Python-скрипта процессора).
     - Отправить пачку сообщений на анализ с промптом, обученным извлекать контекстные даты.
     - При rate limit (например, 429 Too Many Requests) - переключиться на fallback модель из пула.
   - **Календарь:** Если LLM нашла событие (с абсолютным timestamp, вычисленным относительно даты сообщения):
     - Проверить `state.db` на дубликаты.
     - Создать событие в Google Calendar через MCP.
     - Сохранить link в `state.db`.
   - Обновить `processed_message_ids`.

### Этап 5: Документация
- Создать `README.md` с описанием установки, настройки ротации моделей, списка исключений и запуска крона.

## Файлы, которые будут изменены/созданы
- `tg_cal_config.yaml`
- `state.db`
- `check_health.py`
- `processor.py` (или скрипт для крона)
- `README.md`

## Тестирование и валидация
- Проверка ротации: скрипт должен корректно переходить к следующей модели при симуляции 429 ошибки.
- Проверка парсинга: скормить текст "завтра позвонит коллега в три" и убедиться, что создается событие на 15:00 следующего дня (относительно даты самого сообщения).
- Проверка исключений: сообщения из чатов в `excluded_chats` не должны передаваться в LLM.

## Риски и открытые вопросы
- **Telegram MCP:** Готовые MCP для Telegram могут фокусироваться на ботах. Возможна необходимость написания своего MCP-сервера на базе Pyrogram.
- **Rate Limits:** Чтение сообщений из множества чатов может упереться в лимиты Telegram API. Необходим разумный polling interval или webhook (сложнее для user session).
- **Парсинг событий:** Извлечение точного времени и места из свободного текста требует качественного промпта для LLM.
